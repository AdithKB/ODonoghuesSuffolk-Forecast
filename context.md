# O'Donoghues Suffolk Street — Demand Forecasting System
## Project Context

---

## 1. The Problem

O'Donoghues bar at 15 Suffolk Street, Dublin 2 (trading as Markey Pubs, owner Des Markey) runs a busy pub with live music, a full food service, and highly variable footfall driven by weather, sports events, tourism, and the day of the week. The owner currently makes staffing and stock decisions on gut feel and experience.

The cost of getting this wrong is asymmetric:
- **Under-predicting** a busy Friday = understaffed, stockouts, lost revenue, bad customer experience
- **Over-predicting** a quiet Tuesday = wasted labour, food spoilage

This system predicts hourly bar demand (orders count) and food service demand (food tickets count) for any given day, displayed through a live web dashboard so management can plan shifts and stock in advance.

## 5. Feature Engineering (`src/features.py`)

### Lag features
- **Row-based** (`lag_1`, `lag_2`, `lag_3`): previous consecutive bar-open hours. Captures within-shift momentum.
- **Calendar-aligned** (`lag_24h`, `lag_48h`, `lag_168h`, `lag_336h`): exact same clock-hour on prior days. Implemented via index join (not row shift) to handle overnight gaps correctly.
- **Same-slot 4-week average** (`{target}_same_slot_4w_avg`): mean of the same (hour, weekday) slot at T-7d, T-14d, T-21d, T-28d. This is the single strongest anchoring feature — equivalent to "what do we normally do at 13:00 on a Friday?" Ranked #2 by XGBoost gain immediately on introduction.

### Rolling features
Computed on bar-open row sequence, shifted by 1 before applying (no leakage). Windows: 3, 6, 12, 24 hours and 168 hours.

### Calendar features
Hour, weekday, month, quarter, week_of_year, is_weekend, is_friday_saturday. Cyclic sin/cos encodings provided (useful if ever trying linear models). Irish bank holidays via `holidays` library.

### Venue features
Food service windows, breakfast/lunch/dinner windows, live music window flag, hours elapsed since food service closed. These encode the pub's operating structure so the model knows what kind of hour it's predicting.

### External features
Weather (temp_c, rain_mm, wind_speed_kmh), airport arrivals + z-score, cruise ship flags, pedestrian footfall (DCC counters at Suffolk/Nassau St), Fáilte Ireland upcoming events, school holidays, payday periods, college term.

### Interaction features
`weekend_x_music`, `rain_x_weekend`, `event_intensity` (sum of all event flags), `tourism_pressure` (airport z-score + cruise flag).

### Feature set split
- `SAFE_FOR_NEXT_DAY`: features derivable without knowing today's actual orders. Used for next-day forecasting (dashboard use case).
- `FULL_FEATURES`: adds intra-day lags (lag_1/2/3) and current-hour footfall. Would be used for same-day nowcasting (not yet built).

## 6. Model Architecture (`src/model.py`)

### Baseline
Same-hour-last-week (`lag_168h`) with slot rolling-mean fallback. Mirrors what a manager does intuitively. Saved as `baseline_{target}.pkl`. Used as the benchmark to beat and as a sanity check in the dashboard.

### Global XGBoost
Single XGBRegressor trained on all hours. 8-fold walk-forward CV (rolling-origin). Saved as `xgb_{target}.json`. 800 estimators, early stopping on last-20%-of-training validation set.

### Shift-specific XGBoost (added June 2026)
Four shifts per target, tuned separately with Optuna (50 trials, 3-fold walk-forward CV per trial):

| Shift | Hours |
|---|---|
| `lunch` | 12–15 |
| `evening` | 17–20 |
| `late_bar` | 21–02 (wraps midnight) |
| `off_peak` | 09–11, 16 |

Saved as `xgb_{target}_{shift}.json` + `best_params_{target}_{shift}.json`. Dashboard routes each hour to its shift model at prediction time, falls back to global model if shift model absent.

### Asymmetric loss
Custom XGBoost objective in `_asymmetric_obj()`. Under-predictions (pred < true) penalised 2× more than over-predictions, because in a pub context understaffing/stockouts are more costly than having one extra staff member on the floor.

### CV Results (on synthetic data, June 2026)

| Shift | orders_count MAE (before → after) |
|---|---|
| lunch | 8.10 → 4.35 |
| evening | 7.78 → 5.02 |
| late_bar | 6.25 → 3.60 |
| off_peak | 3.35 → 1.41 |

**Critical caveat:** R²=0.838 and all these MAE figures are on synthetic data. The model has learned the rules of the synthetic generation script, not real pub behaviour. When real POS data arrives, expect performance to initially drop significantly before improving as the model learns real signal. The architecture is the value — not the current numbers.

## 8. Dashboard (`dashboard/app.py`)

Deployed at: **https://odonoghuessuffolk-forecast.streamlit.app**

### Sidebar
- **Date picker:** Styled date display (tappable on mobile) with invisible `opacity:0` date_input overlaid. Prev/Next/Today navigation buttons using `on_click` callbacks (avoids Streamlit session_state timing violation).
- **Weather overrides:** Rain and temperature controlled by slider + typeable number_input (synced via callbacks). Auto-populates from real weather data for the selected date.
- **Event toggles:** Live music, Special event, Major sports, Cruise ship. St. Patrick's week is auto-detected (March 10–17) rather than manual — fixed calendar events shouldn't be user-controlled.

### Main panel
- Hourly forecast chart (XGBoost vs baseline, food tickets on secondary axis)
- Narrative briefing card: human-readable summary of the day's signals ("Busy Saturday — live music tonight, Aviva match afternoon"). Rule-based, no LLM.
- Feature drivers panel: top signals influencing today's forecast with importance bars.

### Theme
Dark, monospace/data-forward. Bloomberg terminal aesthetic. `primaryColor = "#F59E0B"` (amber) in `.streamlit/config.toml`. Custom CSS for compact sidebar buttons, hidden slider tick marks, transparent number inputs.

### Data freshness
- GH Actions cron runs `refresh_data.py` every hour at :30 → commits updated data files → triggers Streamlit Cloud redeploy
- Dashboard: `@st.cache_data(ttl=3600)` + `st_autorefresh(interval=3600000)` as belt-and-suspenders

## 10. Key Engineering Decisions and Why

| Decision | Why |
|---|---|
| Synthetic-first approach | No real data available yet. Build and validate the architecture so it's ready on day one. Clearly labelled as synthetic everywhere. |
| XGBoost over neural nets | Dataset is small (~3,800 CV rows). Trees generalise better in low-data regimes. Interpretable feature importance useful for building trust with the owner. |
| Shift-specific models | A single global model regresses to the mean. Lunch demand cares about weather and food; late bar demand cares about weekends and live music. Splitting lets each model specialise. |
| Asymmetric loss (2× under) | Understaffing and stockouts cost more than over-preparedness. The model should lean slightly high on busy hours. |
| `same_slot_4w_avg` | The strongest anchoring signal that isn't just memorising the synthetic script. Equivalent to "what do we normally do at this time?" — the first question any manager asks. |
| Walk-forward CV only | No random splits. Time-series data must be evaluated temporally. Random CV would leak future patterns into training. |
| `on_click` callbacks for date nav | Streamlit ≥1.40 raises `StreamlitAPIException` if you write to a widget-bound session_state key after that widget renders in the same run. Callbacks execute between reruns, before rendering. |
| St. Patrick's auto-detect | Fixed calendar events shouldn't be manual overrides. March 10–17 is always high demand. Only unexpected signals (a surprise event, a private booking) need manual override. |
| Hourly refresh via GH Actions | Streamlit Community Cloud doesn't support server-side cron. GH Actions pushes fresh data to the repo, triggering Streamlit to reload from the updated files. |

## 12. How to Resume This Project

**To retrain after getting real POS data:**
```bash
# 1. Drop real POS CSV at data/synthetic/odonoghues_hourly.csv
#    (columns: timestamp_hour, orders_count, food_tickets_count — see config/schema.yaml)

# 2. Rebuild features
python src/features.py

# 3. Retrain all models (global + 8 shift-specific with Optuna — takes ~15 min)
python src/model.py

# 4. Regenerate EDA charts
python src/run_eda.py

# 5. Push — Streamlit Cloud picks up new models automatically
git add models/ data/ && git commit -m "retrain on real POS data" && git push
```

**To run the dashboard locally:**
```bash
streamlit run dashboard/app.py
```

**To refresh all external data manually:**
```bash
python refresh_data.py
```
